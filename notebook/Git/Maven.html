Maven是什么？

Maven的作用

  • 项目构建：提供标准的、跨平台的自动化项目构建方式。管理整个项目的生命周期。
  • 依赖管理：方便快捷的管理项目依赖的资源(jar包)，避免资源间的版本冲突问题
  • 统一开发结构：提供标准的、统一的项目目录结构

mvn clean

删除target文件夹

mvn compile

重新生成target文件夹

mvn package

mvn package = mvn compile + mvn test + mvn 打包

mvn clean package

mvn clean package = mvn clean + mvn compile + mvn test + mvn 打包

mvn install

将该项目安装到本地仓库，在package后使用

mvn clean 之后记得重新 mvn compile

maven快速构建项目

  • maven构建空项目，然后自己把项目目录标记下。
  • maven构建quick-starter项目，同样对目录稍加修改即可。

maven 依赖传递

  • 最短路径原则：

对于多级依赖出现相同jar的不同版本，maven会选择路径最短的依赖

  • 声明优先原则

对于多级依赖出现相同jar的不同版本，并且所经历的路径相同时，maven会选择最先声明的依赖版本

  • 同级依赖，后加载覆盖先加载

对于同一级的依赖出现相同jar的不同版本，maven会根据依赖声明的先后顺序，选择后声明的依赖版本

pom引依赖 optional选项 标记为可选依赖

optional为true时控制引用该依赖的程序被其他程序依赖时，不传递该依赖。但optional不影响继承。

用exclusion排除不需要的和引用的依赖一并传递过来的子依赖

依赖范围

  • <scope>标签用来设置依赖范围
  • 作用范围
    • 主程序范围有效 (main文件夹范围内)
    • 测试程范围有效 (test文件夹范围内)
    • 是否参与打包 (package指令范围内)
  • 依赖范围的传递
    只有间接依赖打包了，直接依赖被引用时，间接依赖才会被传递
    只有间接依赖<scope>是compile或runtime时，才会被传递

    这张图第一列表示间接依赖的<scope>，第一行表示直接依赖的<scope>
    值表示间接依赖被传递后的依赖范围。

项目构建生命周期

  • Maven对项目构建的生命周期划分为3套
    • clean: 清理工作
    • defafult: 核心工作，例如编译，测试，打包，部署等
    • site: 产生报告，发布站点等



      maven执行到某一生命周期节点，会把之前的生命周期过程全部执行。

生命周期可以理解为是几岁，插件可以理解为是几岁干的那件事

插件

  • 插件与生命周期内的阶段进行绑定，在执行到对应生命周期时就执行对应的插件功能
  • 默认maven在各个生命周期上绑定有预设的功能
  • maven可以通过插件自定义其他功能

    // phase描述的maven运行到的阶段，指定该阶段执行maven-source-plugin插件。
    // <goal>jar<goal>意思为对源码打包。具体看插件使用文档。

Maven 聚合(就是用于多模块多依赖统一维护管理)

试想四个模块逐级依赖，若其中一个模块更新了，另外三个模块并不能知道这个模块更新了，有可能就不好使了。
所以，我们希望一起管理这四个模块(四个工程、四个依赖)
因此，引出了聚合的概念，聚合是在这四个模块之上单独建一个模块(依赖，工程)，对这四个模块一同编译，一同管理

<!--定义该工程用于进行构建管理-->
<packaging>pom</packaging>
<!--管理的工程列表-->
<modules>
    <!--具体的工程名称-->
    <module>模块位置</module>
    <module>模块位置</module>
    <module>模块位置</module>
    <module>模块位置</module>
</modules>
<!--定义该工程用于进行构建管理-->
<packaging>pom</packaging>
<!--管理的工程列表-->
<modules>
    <!--具体的工程名称-->
    <module>模块位置</module>
    <module>模块位置</module>
    <module>模块位置</module>
    <module>模块位置</module>
</modules>

这个模块可以没有代码，即可以没有src目录，这个项目只提供pom文件去做管理

Maven 模块继承

  • 考虑一个问题，D模块依赖A和B，若A模块依赖 C1.0，B模块依赖 C2.0，那么很有可能因为C1.0和C2.0不兼容而出错。

    像图示这种情况ssm_dao依赖5.2.0但是根据路径最短原则，只会使用5.1.9的依赖。但ssm_dao很有可能需要spring-context 5.2.0版本的内容，此时5.1.9就不能兼容5.2.0也就不能兼容ssm_dao模块。

  • 为了避免这种情况，于是我们就想，能不能再造一个新的模块，管理这些重复依赖所使用的版本。

<dependencyManagement>
    <!--具体的依赖及版本-->
    <dependency>
        gav
    </dependency>
</dependencyManagement>
